### Third Body: Governance Distribution (All for One)

The network's collective inference capability serves your personal AI agent at any time. The more you contribute, the more you can invoke, the greater your influence over the network's direction.

This is the dual meaning of All for One: compute returns and governance rights.

**Compute returns.** Your Atman AI agent doesn't rely only on local compute. When it needs complex reasoning, it can invoke global node GPU compute through the Petals distributed inference network — your contribution level in the network determines your invocation priority and bandwidth ceiling. More contribution means a stronger AI agent.

**Governance rights.** Monacelium governance is not one-coin-one-vote, but one-person-one-base-score, with scores calculated from multi-dimensional participation. Specific logic:

Each epoch (7 days) settles once. A node's base score is determined by the geometric mean of four weighted dimensions:

- **Data volume contribution** (weight 30%): bytes of data submitted and ZKP-verified in the epoch
- **Data quality score** (weight 40%): DataDAO contract evaluation of scarcity and diversity — homogeneous data has diminishing marginal value, real life logs far outvalue batch-generated data
- **Node uptime** (weight 20%): proportion of epoch time node remained routable
- **Data uniqueness** (weight 10%): overlap with other network nodes — lower overlap scores higher, preventing copy-paste score farming

Additionally, a Sybil penalty coefficient: once a node is flagged as suspicious, its score for that period goes to zero. Code contributions, bioacoustic data collection, community outreach, and document translation add through Bonus multipliers on top of base score, capped at 1.5× base score.

This formula has one core principle: **let real building behavior, not capital holdings, determine the network's direction.** The formula is open-source, auditable by anyone; DAO can adjust dimension weights through governance vote.

Governance distribution is the last of the three bodies to mature. It requires sufficient user count and sufficiently decentralized node distribution to truly resist the inertia of capital concentration. This is a post-Phase 3 target on the Monacelium roadmap — not a Day 1 promise.

### Layer 1 — Communication: Reticulum

**Core role:** Monacelium's nervous system. All inter-node communication — identity handshakes, data sync, inference request routing — runs on this layer.

**Technical choice: Reticulum Network Stack (RNS)**

RNS is not a TCP/IP replacement — it is a fundamentally different network philosophy. Packets carry no sender IP or location information; destinations are identified by 16-byte public key hashes (Truncated SHA-256). No node needs a complete routing table; each node decides the next hop based only on local neighbor state.

Encryption is enabled by default and cannot be disabled: X25519 elliptic curve key exchange + AES-256-CBC symmetric encryption, with ephemeral key pairs generated per session for Forward Secrecy. Even if long-term private keys leak, past communications cannot be decrypted.

**Dual-plane network topology (V8.1 key correction)**

| Plane | Protocol | Bandwidth | Purpose |
|-------|----------|-----------|---------|
| Control Plane | RNS over LoRa (915/868 MHz) | 0.3–5.5 kbps | Identity handshakes, route broadcasts, emergency beacons, key exchange |
| Data Plane | RNS over WiFi Mesh / Tun | 10–100 Mbps | AI inference, model updates, bulk data sync |

LoRa does not handle LLM inference output (typically 3–5 bytes/token, 5–20 tokens/second — far beyond LoRa bandwidth). This was a V8.0 design error; V8.1 mandates strict control/data plane separation.

**Hardware implementation: RNode**

Flashing ESP32 + LoRa SX1262 module with RNode firmware joins the Reticulum mesh network. SX1262 spot price under $5 USD, ESP32-S3 under $3 USD. Complete RNode assembly cost under $15 USD.

**Contribution entry points:**

- RNode firmware — ESP32-S3 AI instruction set integration, power optimization
- LXMF client — iOS/Android native implementation

### Layer 2 — Hardware: The Vessel / Atman Node

**Core role:** Physical shelter for Monads. All sensor collection, local inference, and encrypted storage happens here.

**Atman Node V1 Specs (BOM target: $85 USD)**

| Component | Model | Function | Est. Cost |
|-----------|-------|----------|-----------|
| MCU (low-power guardian) | ESP32-S3 | 24/7 sensing & wake, TinyML inference | ~$3 |
| SBC (main compute) | Raspberry Pi 5 (8GB) | ZKP generation, local LLM inference | ~$60 |
| Liveness verification | PPG sensor + MEMS mic array | Multimodal biometric collection | ~$10 |
| Communication module | SX1262 LoRa HAT | Reticulum control plane | ~$5 |
| Secure element | ATECC608A | Hardware private key storage, never leaves SE in plaintext | ~$2 |
| Audio decoder | PCM5102 / MAX98357A (I2S) | High-fidelity audio output | ~$3 |
| Storage | MicroSD (encrypted) | Local memory vault, life logs | ~$2 |

**Dual-core architecture logic:**

ESP32-S3 handles 24/7 standby monitoring — extremely low power, continuously collecting environmental signals, waking RPi 5 when trigger conditions are met. RPi 5 handles compute-intensive tasks — ZKP generation, local LLM inference, federated learning gradient computation. They communicate via UART/SPI; ESP32-S3 is the gatekeeper, RPi 5 is the brain.

**TinyML inference (ESP32-S3 layer)**

NCNN framework deploys <10MB quantized models for real-time voiceprint recognition, liveness pre-detection, and wake word detection. Models quantized to INT8; inference latency on ESP32-S3 vector instruction set <50ms.

**Edge LLM inference (RPi 5 layer)**

llama.cpp deploys Llama 3.2 1B Q4_K_M quantized model (~700MB, fits RPi 5 RAM comfortably). Inference speed ~5–15 tokens/second, sufficient for local dialogue, context memory, and data labeling assistance. 3B models run at acceptable latency; 7B+ models too slow on RPi 5, recommended to route to Petals network.

**ATECC608A secure element**

Private keys generated and stored inside the secure element, never exposed as plaintext to the host MCU. All signing operations execute inside the chip; the host can only submit data to sign and receive the signature. Hardware cloning attacks are blocked at the silicon layer. Cost ~$2 USD — the highest-ROI component in the entire security architecture.

### Layer 3 — Security: Liveness Verification & Zero-Knowledge Proofs

**Core role:** Making Monacelium impossible to fool with deepfakes, and making data untraceable to individuals.

**Multimodal liveness verification (V8.1 corrected)**

V8.0 originally specified Laser Doppler Vibrometry (LDV), claiming $5 USD BOM — a severe underestimate. Research-grade LDV equipment costs tens of thousands of dollars; no consumer-grade implementation exists.

V8.1 adopts multimodal fusion:

- **PPG (photoplethysmography):** Detects heart rate waveforms and vascular pulsation patterns — impossible to replicate with speakers
- **MEMS microphone array:** Captures vocal cord vibration characteristics during speech, combined with voiceprint model for liveness detection
- **Facial liveness detection (optional):** Detects blinks, micro-expressions that deepfake models struggle to perfectly replicate

Combined BOM ~$8–15 USD, anti-spoofing effectiveness equal to or exceeding consumer-grade LDV.

**Halo2 zk-SNARK zero-knowledge proofs**

Data on the Atman node is feature-extracted locally within a TEE (Trusted Execution Environment), then Halo2 (Plonk + IPA) generates zero-knowledge proofs. Halo2's core advantage: "Toxic Waste-free" trusted setup — no multi-party ceremony required, anyone can independently verify the circuit, and recursive proof composition is supported.

RPi 5 benchmarks (arXiv:2512.10020):

| Circuit scale | Purpose | RPi 5 time |
|--------------|---------|------------|
| Small (~thousands of gates) | Liveness ZKP, identity handshake | ~1–5 seconds |
| Medium (~tens of thousands) | Physiological feature ZKP | ~5–15 seconds |
| Large (~hundreds of thousands) | zkFL gradient encryption | ~30–60 seconds |

Liveness verification user-facing latency: 1–5 seconds, acceptable. zkFL's 30–60 seconds is acceptable for background non-real-time tasks but requires async handling for real-time flows.

### Layer 4 — Intelligence: Distributed Inference & Collective Intelligence

**Core role:** Let edge devices invoke AI capabilities beyond local compute limits; let global node compute form a decentralized collective brain.

**Petals + Hivemind: Distributed model inference**

ESP32-S3 can't run 70B parameter LLMs. RPi 5 is too slow for 7B+. Petals solves this: splits large language models into "blocks" distributed across global volunteers' consumer GPUs (RTX 3090/4090). Edge nodes only send inference requests; Hivemind coordinates compute routing, returns results.

BitTorrent-like fault-tolerant routing: when a compute node goes offline, the system automatically reroutes to other nodes holding the same model blocks — inference continues uninterrupted. This avoids single-cloud lock-in and strengthens censorship resistance.

**NatureLM-audio: Cross-species sensing**

Integrates Earth Species Project's NatureLM-audio (arXiv:2411.07186). This audio language foundation model's real capabilities: species sound classification, acoustic event detection, soundscape description. It doesn't "understand animal speech" — it is a high-precision ecological acoustics classification engine, capable of recognizing acoustic signatures of thousands of species, with zero-shot generalization to unseen species.

Deployed on outdoor Atman nodes, it continuously collects environmental soundscapes; classified data enters ecological DataDAOs for conservation research institutions to subscribe to.

**Federated learning pipeline (zkFL)**

```
Local training (differential privacy fine-tuning)
    ↓
Gradient encryption (homomorphic encryption / ZKP protection)
    ↓
Federated aggregator (trusted aggregator — short-term interim)
    ↓
Global model update
    ↓
Distributed to nodes via Reticulum WiFi Mesh
```

Note: Full zero-knowledge gradient encryption generates in 30–60 seconds on RPi 5 for large circuits. Short-term uses "trusted aggregator + differential privacy (ε=0.1)" as fallback; upgrades to full zkFL in Phase 3+ as SNARK hardware accelerators mature.

### Flywheel Logic

One sentence mechanism: **You contribute data → AI companies purchase access proofs with tokens → 90% of tokens return to contributing nodes → 10% is permanently burned → data never leaves your device, but its value flows through the network.**

The full flywheel: You contribute data → data is purchased by AI companies → tokens return to you → tokens incentivize more contribution → data quality and scale improve → AI companies pay more.

Each rotation increases flywheel momentum. This is not an artificially subsidized loop — its driving force comes from AI companies' structural demand for high-quality training data. This demand in 2026 is structural, not cyclical.

---

### Two Tokens, One Clear Division

**MONA** is the protocol's native governance and utility token. Hold it and you have a voice. Use core network functions and you consume it. Total supply is fixed at 1 billion, hardcoded in the genesis contract; any issuance requires 75% DAO vote approval and a 365-day lockup buffer. No exceptions.

**DAT** is the data asset token issued by each DataDAO independently. MusBioDAO has its MBIO, health data DataDAO has its token, ecological acoustics DataDAO has another. Contribute to a data type, mint that DataDAO's DAT. DAT represents your ownership and governance rights over a specific dataset; AI companies must purchase and burn it to access data.

The design principle: isolate volatility. MONA's value comes from the entire protocol's network effects; DAT's value comes from specific data type market demand. They interact but don't hold each other hostage.

---

### Data Doesn't Move. Value Does.

This is the most important principle in the entire model.

Your raw data — consumption records, physiological signals, environmental acoustics, movement patterns — never leaves your Atman device. Monacelium uses Halo2 zk-SNARK to generate zero-knowledge proofs locally: mathematical proofs that tell the network "this data is real, meets quality standards, comes from a real human node" — without revealing the data's content.

AI companies subscribe to the "proof," not raw data. They receive verified training signals; you retain privacy. This isn't a trade-off between privacy and value — it's achieving both simultaneously through cryptography.

**Passive income: Accumulating while the node runs.**

Your Atman node online simultaneously does three valuable things: routing relay for Reticulum, contributing edge inference compute, and collecting and locally processing environmental data. All three convert to participation base scores; scores determine DAT minting allocation per epoch (7 days).

**Active income: You earn when data is purchased.**

When AI companies query your DataDAO via API: 10% DAT permanently burned (reducing supply, protecting scarcity), 90% DAT distributed proportionally to all node contributors, 1% MONA burned (part of protocol fees).

With AI training data demand continuously expanding, there is upside potential, but we make no unverifiable promises.

### Scarcity Protection Mechanisms

Three safeguards keep the flywheel spinning:

**Declining emission curve.** MONA mining pool uses annually decreasing release rate: Year 1-2 releases 0.03%/day (cap 821,917 MONA/day), Year 3-5 drops to 0.015%, Year 6+ drops to 0.005%. Early contributors receive highest density incentives as compensation for building sparse network.

**Automatic buyback and burn.** 50% of every data transaction fee enters the Protocol Liquidity Treasury. System checks every 24 hours: when MONA price falls below 95% of 30-day moving average, automatic buyback and burn executes. Projected that after Year 3, annualized burn rate exceeds annualized issuance rate — net circulating supply begins declining.

**Unstaking flow control.** Maximum 0.5% of total staked MONA unlockable per day network-wide; requests exceeding cap automatically queue; emergency unstaking requires burning 5% of staked amount as penalty fee. This prevents panic-selling from catastrophically impacting liquidity.

### Harberger Tax: Limited but Precise Application

Monacelium implements Harberger Tax, but strictly scoped.

Principle: if you hold a scarce network resource, you must publicly price it and pay proportional tax. Anyone willing to pay your listed price can immediately acquire it. This forces resources to flow toward whoever can create the most value, preventing early occupants from hoarding indefinitely.

**Harberger Tax applies only to three network scarce infrastructure resources:** high-traffic relay node priority bandwidth rights, scarce RNS address space in the network, specific DataDAO unique namespaces.

**It explicitly does not apply to:** your personal data, your device, any personal private property. Personal data uses a "voluntary licensing model" — you decide whether to authorize; no one can force-purchase your data.

**AI company scenario.** An AI research institution needs real daily life logs from Taiwan users to fine-tune an embodied AI world model. It subscribes via DataDAO API; each query burns MONA and DAT. 90% of burned DAT is distributed proportionally to all nodes that contributed to this dataset. Your Atman node received a distribution while you slept.

OpenAI, Google, and Anthropic collectively hit the same wall in 2024–2026: high-quality training data exhaustion.
